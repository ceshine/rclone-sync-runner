# Technical Design Document: MVP `rclone-sync-runner`

- Date: 2026-02-26
- Target package: `rclone-sync-runner`
- Runtime context: `systemd` periodic jobs

## 1. Overview

Build a Python package that executes multiple `rclone sync` tasks defined in a YAML file.
Each run should produce clear, structured logs and a final summary indicating which folder pairs succeeded or failed.

This MVP emphasizes reliability, observability, and simplicity. Notification integrations (for example Telegram) are intentionally deferred but designed for easy addition.

## 2. Goals

1. Read a list of sync jobs from YAML.
2. Execute `rclone sync` sequentially for each job.
3. Parse `rclone` JSON log events using `orjson`.
4. Emit informative logs for:
   - Job start and end
   - Progress/stat events
   - Failure details
   - Run-level summary
5. Exit with non-zero status when any job fails (systemd-friendly).

## 3. Non-Goals (MVP)

1. Concurrent execution of multiple sync jobs.
2. Retry/backoff policies.
3. Built-in Telegram or other notification channel implementations.
4. Stateful persistence database for historical runs.
5. Managing `rclone` lifecycle via `rclone rcd`.

## 4. Assumptions and Constraints

1. `rclone` is installed and available on `PATH` in the service environment.
2. YAML config path is provided by CLI argument.
3. Jobs are independent and can run sequentially.
4. `rclone` credentials/remotes are already configured externally.

## 5. Configuration Design

## 5.1 YAML Schema (MVP)

```yaml
version: 1
global:
  rclone_bin: rclone
  log_level: INFO
  continue_on_error: true
jobs:
  - name: photos_backup
    source: /data/photos
    destination: remote:backups/photos
    extra_args:
      - --delete-during
      - --fast-list
  - name: docs_backup
    source: /data/docs
    destination: remote:backups/docs
```

## 5.2 Validation Rules

1. `version` must be supported (`1` for MVP).
2. `jobs` must be non-empty.
3. Each job requires `name`, `source`, `destination`.
4. Job names must be unique.
5. `extra_args` is optional and must be a list of strings.

Validation approach: Pydantic models for strict input validation and clear error messages.

## 6. Package Structure

```text
rclone_sync_runner/
  __init__.py
  cli.py
  config.py
  models.py
  runner.py
  rclone_subprocess.py
  logging_setup.py
  notifiers/
    __init__.py
    base.py
    logging_notifier.py
```

### Module Responsibilities

1. `cli.py`
   - Typer entrypoint.
   - Accepts config path and run options.
2. `config.py`
   - Load YAML via `pathlib.Path`.
   - Validate into typed models.
3. `models.py`
   - Data models for job config and run results.
4. `rclone_subprocess.py`
   - Build command args and run subprocess.
   - Stream stderr lines for event parsing.
5. `runner.py`
   - Orchestrate per-job execution and run summary.
6. `logging_setup.py`
   - Standard logging initialization.
7. `notifiers/*`
   - Interface for future channels; logging notifier used in MVP.

## 7. Execution Flow

1. CLI loads config.
2. CLI validates config; fail fast if invalid.
3. For each job:
   - Build `rclone sync` command.
   - Start subprocess.
   - Parse stderr JSON lines with `orjson.loads`.
   - Track job stats and errors.
   - Determine success from process exit code.
4. Emit run summary:
   - total jobs
   - succeeded
   - failed
   - duration
5. Exit code:
   - `0` if all jobs succeed
   - `1` if one or more jobs fail
   - `2` for config/usage/runtime orchestration errors

## 8. `rclone` Invocation and Parsing

## 8.1 Command Template

```bash
rclone sync <source> <destination> \
  --use-json-log \
  --log-level <level> \
  --stats 30s \
  --stats-log-level INFO \
  <extra_args...>
```

## 8.2 Parsing Strategy

1. Read subprocess `stderr` as text stream (line-by-line).
2. For each line:
   - Attempt `orjson.loads(line)`.
   - On parse failure, log raw line as unstructured fallback.
3. Interpret event fields:
   - `level=error` -> capture error event
   - `stats` present -> update latest stats snapshot
4. Keep bounded in-memory error samples (for example first 20 events) to avoid unbounded memory growth.

## 8.3 Why `orjson`

1. Faster JSON decode for log-stream workloads.
2. Strict behavior and lower overhead than stdlib `json`.
3. Aligns with explicit project requirement.

## 9. Logging and Result Model

## 9.1 Logging Principles

1. Use module-level logger:
   - `LOGGER = logging.getLogger(__name__)`
2. Structured, concise log messages.
3. Include job context (`job_name`, `source`, `destination`) on all relevant messages.

## 9.2 Result Objects

`JobRunResult` (per job):

1. `job_name: str`
2. `source: str`
3. `destination: str`
4. `started_at: datetime`
5. `ended_at: datetime`
6. `duration_seconds: float`
7. `return_code: int`
8. `succeeded: bool`
9. `error_count: int`
10. `error_samples: list[str]`
11. `last_stats: dict[str, Any] | None`

`RunSummary` (whole execution):

1. `total_jobs: int`
2. `successful_jobs: int`
3. `failed_jobs: int`
4. `duration_seconds: float`
5. `results: list[JobRunResult]`

## 10. Notification Extension Point (Future-ready)

Define a notifier interface now:

```python
class Notifier(Protocol):
    def on_run_finished(self, summary: RunSummary) -> None:
        ...
```

MVP wires `LoggingNotifier` only. Future `TelegramNotifier` can implement the same interface without changing runner flow.

## 11. Systemd Integration

## 11.1 Service Unit (Example)

```ini
[Unit]
Description=Rclone sync runner
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
WorkingDirectory=/opt/rclone-sync-runner
ExecStart=/opt/rclone-sync-runner/.venv/bin/rclone-sync-runner run --config /etc/rclone-sync-runner/jobs.yaml
User=backup
Group=backup
```

## 11.2 Timer Unit (Example)

```ini
[Unit]
Description=Run rclone sync periodically

[Timer]
OnCalendar=hourly
Persistent=true
RandomizedDelaySec=300

[Install]
WantedBy=timers.target
```

Behavioral expectation: systemd/journald captures stdout/stderr logs, and non-zero exit status marks failed runs for monitoring.

## 12. Testing Strategy

1. Unit tests for YAML config validation (valid/invalid cases).
2. Unit tests for command construction.
3. Unit tests for JSON line parsing with `orjson`:
   - valid event
   - malformed line fallback
   - error event extraction
4. Unit tests for runner summary and exit-code behavior.
5. Integration-style test with mocked subprocess responses (no real cloud access).

Test tooling:

1. `pytest`
2. Fixtures for temporary YAML files (`tmp_path`).
3. Mock `subprocess.Popen` and streamed stderr lines.

## 14. Risks and Mitigations

1. Risk: malformed/non-JSON rclone output line.
   - Mitigation: fallback unstructured logging, continue processing.
2. Risk: long-running jobs with large error output.
   - Mitigation: bounded error samples in memory.
3. Risk: credentials/environment mismatch in systemd.
   - Mitigation: explicit service `User`, environment checks, clear startup logs.
